## 2. Admin UX Review

Admin is 8 tabs (Geral, Anamnese, Biomarcadores, Plano alimentar, SuplementaĂ§ĂŁo, Consultas, RelatĂłrio, Chat) editing one `users/{uid}` doc, with a genuinely solid infra layer already: cursor-paginated client list, `getCountFromServer()` aggregate stats (no full-collection reads), per-tab save timestamps + a `historicoAlteracoes` audit array, dirty-state indicators. This part is production-grade.

The workflow itself is where the manual burden lives:

1. **Biomarker entry is fully manual per row** â€” name, value, unit, and *four separate gender-split range pairs* (population/optimal/risk/action), typed per client, per test. There's an "Auto" option for the interpretation tag (dropdown at `admin.html:1900-1905`) that can defer to the dashboard's `autoTag()` logic â€” good â€” but **`delta` is a free-text field the clinician types by hand** (`admin.html:1906`) even though the previous value sits one Firestore read away in `users/{uid}/analises/`. This is the single biggest "obviously should be automatic" gap.
2. **No reference-range library** â€” ranges aren't reusable across clients; every new client starts from a blank table. A thyroid panel typed correctly once should never need re-typing.
3. **Nutrition/supplement authoring is 100% free text** â€” meal alternatives are raw textareas, supplement rationale is a textarea. No templates, no reuse across clients with the same `objectivo`.
4. **No priority/severity/follow-up metadata exists anywhere** â€” confirmed by direct search: zero matches for `prioridade`, `importan`, `followUp`, `adesao`, `severidade`, `target` in admin.html. The only quasi-priority signal is the `destaque` (highlight) checkbox on biomarkers, which is presentational, not clinical.
5. **State transitions are one-client-at-a-time** â€” no bulk "advance this cohort" action, which matters once dozens of clients hit the same 6-month reassessment window simultaneously.
6. **Consultation invites are manual click-through** â€” admin manually taps a month grid and time slots per client, no default/suggested slot generation.

## 8. Automation Opportunities

Ranked by effort-to-value, all rule-based (no ML/LLM needed given the data is already structured):

1. **Auto-delta from `analises/` history** â€” trivial, biggest clinician-time win, zero clinical risk (it's arithmetic, not judgment).
2. **Auto-priority suggestions** â€” simple rules over data already entered: any biomarker with `tag=critico` â†’ suggest "Repita este marcador em 6 semanas" priority; any biomarker `tag=atencao` matched against a nutrition-insight keyword â†’ suggest "Aumente a ingestĂŁo de fibra." This reuses the exact `INSIGHT_DEFS`/keyword-matching pattern already built for meal tags (`dashboard.html:4857-4873`) â€” same mechanism, new target.
3. **Stale-notification nudges** â€” `emailFalhouEm` already exists for failed auto-emails (`admin.html:2833`); surface it as a dashboard-style admin alert list instead of requiring the clinician to notice per-client.
4. **Reference-range templates** â€” biggest reduction in manual typing (Â§5.1), moderate effort (needs a small template CRUD + apply-to-client action).
5. **Bulk cohort state transitions** â€” moderate effort, high value once client volume grows past ~30-50 concurrent reassessments.
6. **Consultation slot suggestions** â€” nice-to-have, lower priority than the above since current manual flow, while tedious, isn't error-prone.

## 9. Information Architecture

Current state is genuinely sound for the current scale: single `users/{uid}` doc with subcollections, field-locked security rules, atomic client-ID counter, cursor pagination + aggregate counts avoiding full scans, audit trail via `historicoAlteracoes` + `eliminacoesRGPD`. This isn't MVP-fragile â€” it's a reasonable production data model already.

What changes as the product matures:
- **`prioridades[]` should be a subcollection**, not an array on `users/{uid}` â€” priority items need independent timestamps/dismissal state and will grow unboundedly across cycles, unlike the bounded `biomarcadores`/`suplementacao` arrays.
- **Multi-clinician support** needs `clinicoAtribuido` (Â§4) plus a rule addition in `firestore.rules` scoping which admin can edit which client, if/when Lyvoo hires beyond the current single-admin model â€” not needed today, but the field costs nothing to add now and avoids a migration later.
- **Future AI integration** (e.g., LLM-assisted report drafting) is viable specifically *because* biomarker/anamnese data is already structured JSON rather than free text â€” the one gap is that meal/supplement rationale remains free text, which limits any future automated cross-client pattern analysis. Not urgent; only matters if that's an actual roadmap item.
- **Auditability** is already decent (`historicoAlteracoes`, `eliminacoesRGPD`, `webhookErrors`) â€” no changes needed there.
